    🎯 Обзор¶

    Этот гайд описывает миграцию Quote Calculator с file-based storage на SQLite database.

    Текущее состояние (File Storage)¶

    estimate/
  ├─ client1_2025-10-20_5pax_abc123.json
  ├─ client2_2025-10-21_10pax_def456.json
  └─ ...

backup/
  ├─ abc123.json
  ├─ def456.json
  └─ ...

catalog/
  └─ catalog.json

    Целевое состояние (SQLite)¶

    db/
  └─ quotes.db  ← Один файл для всех данных

    ✨ Преимущества миграции¶

    Решает текущие проблемы¶

    Проблема File Storage SQLite
    Рассинхронизация estimate/ и backup/ ❌ Dual save может упасть ✅ ACID транзакции
    Race conditions в autosave ⚠️ Guard flags ✅ Optimistic locking
    Concurrent edits ❌ Нет защиты ✅ Version tracking
    Undo/Redo ❌ Нет ✅ Audit log готов
    Поиск 🐌 Медленно ⚡ Индексы
    Backup 📁 Копирование папок 📦 Один файл

    Дополнительные преимущества¶

    • ✅ Простой деплой - один файл БД вместо тысяч JSON файлов
    • ✅ Атомарные операции - нет partial writes
    • ✅ Data integrity - foreign keys, constraints
    • ✅ Upgrade path - легко мигрировать на PostgreSQL позже
    • ✅ Performance - индексы для быстрого поиска

    📦 Требования¶

    Системные требования¶

    • ✅ Node.js >= 18.0.0
    • ✅ ~50MB свободного места (для БД)
    • ✅ Права на запись в директории db/

    Подготовка¶

    # 1. Убедитесь что у вас последняя версия кода
git pull origin main

# 2. Установите зависимости (включая better-sqlite3)
npm install

# 3. Создайте backup текущих данных
cp -r estimate/ estimate_backup/
cp -r backup/ backup_backup/
cp -r catalog/ catalog_backup/

# 4. Проверьте что backup создан
ls -la *_backup/

    🚀 Пошаговая инструкция¶

    Фаза 1: Подготовка (5 минут)¶

    1.1. Проверка текущего состояния¶

    # Запустите текущий сервер
npm start

# В другом терминале проверьте health check
curl http://localhost:3000/health

# Проверьте количество смет
ls -1 estimate/*.json | wc -l

    1.2. Остановка сервера¶

    # Ctrl+C в терминале с сервером
# ИЛИ
pkill -f "node server.js"

    Фаза 2: Миграция данных (10-30 минут)¶

    # Проверить что будет сделано БЕЗ реального импорта
npm run migrate:dry-run

    2.2. Реальная миграция¶

    # Запустить миграцию с валидацией

    Фаза 3: Тестирование SQLite (15 минут)¶

    3.1. Запуск с SQLite storage¶

    # Создайте .env файл
cp .env.example .env

# Отредактируйте .env
nano .env

    Установите в .env: 

    STORAGE_TYPE=sqlite
DUAL_WRITE_MODE=false

    3.2. Запуск нового сервера¶

    # Замените server.js на новую версию
mv server.js server-old.js
cp server-with-db.js server.js

# Запустите сервер
npm start

    Ожидаемый вывод: 

    Storage configuration:

    3.3. Тестирование функционала¶

    # Health check
curl http://localhost:3000/health | jq .

# Список смет
curl http://localhost:3000/api/estimates | jq '.estimates[] | {filename, clientName}'

# Загрузить одну смету
curl http://localhost:3000/api/estimates/client1_2025-10-20_5pax_abc123.json | jq .

    3.4. Тестирование в браузере¶

    1. Откройте http://localhost:3000
    2. Проверьте что сметы загружаются
    3. Откройте существующую смету
    4. Внесите изменения и сохраните
    5. Создайте новую смету
    6. Проверьте что все работает

    Фаза 4: Dual-Write режим (Опционально, 1 неделя)¶

    Если хотите максимальную безопасность, используйте dual-write режим:

    # .env
STORAGE_TYPE=sqlite
DUAL_WRITE_MODE=true

    В этом режиме все изменения пишутся и в SQLite, и в файлы. Это позволяет: - ✅ Быстро откатиться на file storage - ✅ Проверить целостность данных - ✅ Постепенно перейти на SQLite

    Рекомендуется: Работать в dual-write режиме 3-7 дней перед полным переходом.

    Фаза 5: Финализация (5 минут)¶

    5.1. Отключение dual-write¶

    После успешного тестирования:

    # .env
STORAGE_TYPE=sqlite
DUAL_WRITE_MODE=false

    5.2. Backup базы данных¶

    # Создайте backup БД
cp db/quotes.db db/quotes_$(date +%Y%m%d).backup.db

# Настройте автоматический backup (опционально)
# Добавьте в crontab:
# 0 3 * * * cp /path/to/db/quotes.db /path/to/backups/quotes_$(date +\%Y\%m\%d).backup.db

    5.3. Архивирование старых файлов¶

    # Создайте архив старых файлов
tar -czf legacy_file_storage_$(date +%Y%m%d).tar.gz estimate/ backup/ catalog/

# Переместите в архив
mkdir -p archive/
mv legacy_file_storage_*.tar.gz archive/

# Опционально: удалите старые директории
# rm -rf estimate/ backup/ catalog/

    🔄 Откат (Rollback)¶

    Если что-то пошло не так, вы можете откатиться на file storage:

    Быстрый откат (если dual-write был включен)¶

    # 1. Остановите сервер
pkill -f "node server.js"

# 2. Верните старый server.js
mv server-old.js server.js

# 3. Удалите .env (используется дефолтный file storage)
rm .env

# 4. Запустите сервер
npm start

    Полный откат (если dual-write не был включен)¶

    # 1. Остановите сервер
pkill -f "node server.js"

# 2. Восстановите backup
rm -rf estimate/ backup/ catalog/
cp -r estimate_backup/ estimate/
cp -r backup_backup/ backup/
cp -r catalog_backup/ catalog/

# 3. Верните старый server.js
mv server-old.js server.js

# 4. Удалите .env и БД
rm .env
rm -rf db/

# 5. Запустите сервер
npm start

    ❓ FAQ¶

    Q: Сколько времени займет миграция?¶

    A: Зависит от количества данных: - 10 смет: ~1 минута - 100 смет: ~2-3 минуты - 1000 смет: ~10-15 минут

    Q: Будут ли работать старые JSON файлы?¶

    A: Да, вы можете импортировать JSON файлы как обычно через UI. Они автоматически сохранятся в SQLite.

    Q: Можно ли вернуться на file storage?¶

    A: Да, в любой момент через .env: 

    STORAGE_TYPE=file

    Q: Увеличится ли размер хранилища?¶

    A: Нет, SQLite обычно эффективнее: - File storage: ~N файлов × (размер + metadata) - SQLite: один файл, сжатие, без дублирования

    Пример: 100 смет - Files: ~15MB (разрозненные) - SQLite: ~8MB (единый файл)

    Q: Как мигрировать на PostgreSQL в будущем?¶

    A: Архитектура готова: 1. Реализовать PostgresStorage (аналогично SQLiteStorage) 2. Экспортировать данные из SQLite 3. Импортировать в PostgreSQL 4. Установить STORAGE_TYPE=postgres

    Q: Что делать с ошибками миграции?¶

    A: Смотрите migration_report.json: 

    cat migration_report.json | jq '.errors'

    Частые ошибки и решения в секции Troubleshooting.

    🔧 Troubleshooting¶

    Ошибка: "better-sqlite3 not installed"¶

    # Решение:
npm install better-sqlite3

# Если ошибка persist, удалите node_modules и переустановите
rm -rf node_modules package-lock.json
npm install

    Ошибка: "EACCES: permission denied, mkdir 'db'"¶

    # Решение: дайте права на запись
mkdir -p db
chmod 755 db

    Ошибка: "Database is locked"¶

    # Решение: остановите все процессы, использующие БД
pkill -f "node server.js"

# Удалите lock файлы
rm -f db/*.db-shm db/*.db-wal

# Перезапустите сервер
npm start

    Некоторые сметы не мигрировались¶

    # Проверьте отчет
cat migration_report.json | jq '.stats.estimates'

# Посмотрите ошибки
cat migration_report.json | jq '.errors[] | select(.type == "estimate")'

# Исправьте проблемные JSON файлы и перезапустите миграцию
npm run migrate:run

    Optimistic locking conflicts¶

    Если видите ошибки "Concurrent modification detected":

    // В UI обработайте 409 ответы:
try {
    await apiClient.saveEstimate(filename, data);
} catch (err) {
    if (err.code === 'CONFLICT') {
        // Предложите пользователю reload
        if (confirm('Данные были изменены. Перезагрузить?')) {
            location.reload();
        }
    }
}

    📊 Checklist¶

    Используйте этот checklist для отслеживания прогресса:

    Подготовка¶

    • Backup текущих данных создан
    • npm install выполнен успешно
    • Текущий сервер работает корректно

    Миграция¶

    • Dry run завершен без ошибок
    • Реальная миграция завершена
    • migration_report.json проверен
    • Все сметы мигрировали (failed = 0)

    Тестирование¶

    • Сервер запускается с STORAGE_TYPE=sqlite
    • Health check проходит
    • Список смет загружается
    • Открытие сметы работает
    • Сохранение работает
    • Создание новой сметы работает
    • Каталог работает

    Финализация¶

    • Dual-write отключен
    • Backup БД создан
    • Старые файлы архивированы
    • Документация обновлена

    📞 Поддержка¶

    3. Откатитесь на file storage если критическая проблема

    4. Создайте issue с:

    5. migration_report.json
    6. Логи сервера
    7. Описание проблемы

    После успешной миграции вы получите:

    • ✅ Единый файл БД вместо тысяч JSON файлов
    • ✅ ACID транзакции - нет рассинхронизации
    • ✅ Optimistic locking - защита от concurrent edits
    • ✅ Audit log ready - готовность к undo/redo
    • ✅ Performance - быстрый поиск через индексы
    • ✅ Upgrade path - легко мигрировать на PostgreSQL

    Время миграции: 30-60 минут (включая тестирование) Риски: Минимальные (есть rollback) Рекомендация: ✅ Выполнить миграцию
